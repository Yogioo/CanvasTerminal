# 决策节点（人工校验分流）MVP 方案

> 状态：讨论结论（待实现）
> 
> 目标：先做最小可用版本，解决“人工确认后分流”连续性问题。

## 1. 背景与问题

当前流程里，人工确认通常依赖：
- 终端里手动输入 `canvas done --route ...`，或
- 右键文本节点“完成并传递”。

问题是交互不连续、易出错、可见性弱。

## 2. MVP 目标

新增一种可创建的节点类型：**决策节点**（定位是人工决策分流节点）。

MVP 必须满足：
1. 可通过右键菜单创建（与文本/终端一致）。
2. 可右键配置按钮列表（按钮文案 + 事件名）。
3. 节点可接收上游消息并在节点内展示。
4. 用户点击按钮后，按按钮事件名匹配连线并把接收到的消息传给下游。

## 3. 非目标（MVP 不做）

- 不做自动决策规则引擎（纯人工点击分流）。
- 不做权限系统。
- 不做复杂队列管理（先按“最新一条消息”处理）。
- 不做超时自动分支。
- 不做审计检索页面（仅保留基础元信息即可）。

## 4. 核心概念

- **事件名（event_key）**：按钮绑定的机器可读 key，用于匹配出边。
- **连线路由键（route_key）**：现有边属性；MVP 中约定其语义与 `event_key` 对齐。
- **待处理消息（pending message）**：决策节点当前展示并等待人工确认的输入内容。

## 5. 交互设计（MVP）

### 5.1 创建节点

画布右键菜单新增：
- 创建节点 / 决策节点

### 5.2 配置节点

决策节点右键菜单新增：
- 编辑按钮配置

配置项结构（每个按钮）：
- `label`（展示文案，如“通过”“驳回”）
- `event_key`（如 `approve` / `reject`）

校验规则：
- `label` 非空
- `event_key` 非空
- 同一节点内 `event_key` 唯一

### 5.3 节点展示

节点内显示：
1. 当前待处理消息（最新一条）
2. 按钮区（按配置动态渲染）

### 5.4 点击按钮后的行为

点击按钮 `event_key = X` 时：
1. 查找当前节点所有出边中 `route_key == X` 的边。
2. 找到目标下游节点。
3. 将当前待处理消息转发给目标下游节点。
4. 清空当前待处理消息（或标记为已处理）。

异常处理：
- 未找到匹配边：toast 提示，不丢消息。
- 当前无待处理消息：按钮禁用。

## 6. 数据与转发语义（MVP）

### 6.1 决策节点数据（建议）

- `title`（可复用现有标题编辑逻辑）
- `buttons: [{ label, event_key }]`
- `pending_message: Option<String>`（仅保留最新一条）

### 6.2 转发内容

MVP 先转发原始消息文本；可附带轻量元信息：
- `chosen_event_key`
- `chosen_label`
- `timestamp_ms`

> 若下游当前只支持纯文本注入，则元信息可先不下发，只在内部日志保留。

## 7. 可用性与可拓展性评价

### 可用性
- 人工确认入口就在节点上，减少心智切换。
- 避免在终端输入命令造成拼写错误。
- 分支关系通过“按钮 + route_key 连线”直观表达。

### 可拓展性
- 后续可平滑扩展：备注必填、超时分支、批量处理、权限控制。
- 设计与现有 route_key 机制一致，不需要推翻现有协议。

## 8. MVP 验收标准

1. 可创建决策节点。
2. 可配置至少两个按钮（如 approve/reject）。
3. 决策节点能接收并显示上游消息。
4. 点击 approve 只走 route_key=approve 的边。
5. 点击 reject 只走 route_key=reject 的边。
6. 无匹配边时有明确提示，且消息不丢失。

## 9. 建议实施顺序

1. 增加 NodeKind + 持久化结构。
2. 增加右键创建入口。
3. 增加右键按钮配置入口与编辑 UI。
4. 增加消息接收/显示与按钮点击转发。
5. 补最小自动化测试（创建、配置、分流、异常提示）。
