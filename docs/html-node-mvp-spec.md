# HTML 节点 MVP 规范（直接可交互方案）

## 1. 目标

HTML 节点的目标是：

- 用户可以直接使用 HTML/CSS 构建节点内容
- 节点中的按钮、输入框、滚动区域等内容应当 **直接可交互**
- 尽量复用浏览器现成的 HTML/CSS/交互能力
- 在无限 Canvas 中尽可能保持较好的性能
- 在保证交互的前提下，把代码复杂度和维护成本控制在可接受范围内

本方案明确拒绝“纯快照节点需要先双击再交互”的体验。

用户看到按钮，就应该能直接点。

---

## 2. 核心结论

HTML 节点第一版不采用以下两种极端方案：

### 不采用方案 A：纯快照优先

原因：

- 快照本身不可交互
- 用户需要先进入节点再点按钮，体验不符合预期
- 节点中的按钮会变成“假按钮”

### 不采用方案 B：所有 HTML 节点永久 live

原因：

- 多节点场景下性能压力太大
- 浏览器实例数量、输入焦点、缩放同步、层级裁剪都会显著增加复杂度
- 维护成本高

---

## 3. 最终采用的方案

HTML 节点第一版采用：

# **单活跃 live + 点击即活 + 首次点击透传 + 非活跃冻结**

含义如下：

1. 节点默认显示网页内容的最近状态
2. 用户点击节点内部任意位置时，不需要先进入特殊模式
3. 系统会自动让该节点成为当前 live 节点
4. 这一次点击事件会继续投递给网页内容
5. 对用户来说，效果应当等价于“直接点中了网页里的按钮”
6. 同时只允许 1 个 HTML 节点处于 live 状态
7. 非活跃 HTML 节点保持冻结显示，以控制性能

这是当前目标下最平衡的方案：

- 保住交互体验
- 控制性能成本
- 尽量复用浏览器逻辑
- 避免把系统做成“多浏览器并发平台”

---

## 4. 用户体验原则

### 4.1 直接点击原则

用户对 HTML 节点的体验应当像这样：

- 看到按钮，直接点按钮
- 看到输入框，直接点进去输入
- 看到可滚动区域，直接滚动

而不是：

- 先双击进入节点
- 再点击里面的按钮

### 4.2 无感激活原则

虽然底层存在“冻结态”和“live 态”，但这种状态切换应尽量对用户无感。

理想体验：

- 用户第一次点击节点内按钮时
- 节点后台自动切换为 live
- 当前点击事件继续进入网页
- 用户感觉不到中间的切换过程

### 4.3 单活跃节点原则

同一时刻只允许一个 HTML 节点保持完整交互态。

原因：

- 大多数用户不会同时操作多个网页节点
- 当前真正活跃交互的节点通常只有一个
- 这是性能和维护成本可控的关键边界

---

## 5. 节点状态模型

每个 HTML 节点在运行时有两种主要状态。

## 5.1 Frozen（冻结态）

冻结态特点：

- 节点显示最近一次网页画面
- 外观上仍然像网页节点
- 不维持完整 live 交互能力
- 资源占用应尽量低

冻结态的目的：

- 降低多节点并存时的性能压力
- 保持画布拖拽、缩放、平移流畅

说明：

- Frozen 可以是快照图像
- 也可以是“保留最近画面但暂停交互”的低成本形式
- MVP 不强制要求内部实现细节，但对用户来说应表现为“节点依然长得像网页”

## 5.2 Live（活跃态）

live 态特点：

- 节点为真实可交互网页
- 可点击、可输入、可滚动、可 hover
- 接收真实浏览器事件
- 资源占用高于冻结态

限制：

- 同时最多 1 个 live HTML 节点

---

## 6. 交互规则

## 6.1 点击即活

当用户点击一个当前非 live 的 HTML 节点内部区域时：

1. 系统将该节点切换为当前 live 节点
2. 若此前已有其他 live HTML 节点，则先将其冻结
3. 将本次点击坐标投递给新 live 的网页内容
4. 按钮、链接、输入框应直接响应该次点击

这条规则是整个方案成立的关键。

## 6.2 首次点击透传

为了实现“按钮直接可点”，必须支持：

- 用户第一次点击节点中的网页元素
- 系统在节点切为 live 后
- 仍然将这次点击传递给网页

否则用户就会感知到：

- 第一次点击只是“激活节点”
- 第二次点击才真正点中按钮

这是不可接受的。

## 6.3 输入焦点

当节点进入 live 状态后：

- 输入框可获得焦点
- 键盘事件应进入该网页节点
- 失去活跃状态后，网页焦点应被释放

## 6.4 滚动行为

当鼠标位于 live HTML 节点内部时：

- 优先交给网页内部滚动区域处理
- 若网页无可滚动内容，可再回退到画布层滚动逻辑（后续可优化）

MVP 首版可先只保证：

- live 节点内部滚动正常

---

## 7. 性能原则

## 7.1 性能目标

在保证“当前节点直接可交互”的前提下，尽量做到：

- 多个 HTML 节点可同时存在于画布中
- 画布拖拽、缩放、平移不明显卡顿
- 非活跃 HTML 节点尽量低成本

## 7.2 性能边界

本方案的性能依赖一个关键约束：

# **同时只有一个 live HTML 节点**

如果未来放开多个 live 节点：

- 性能压力会显著上升
- 维护复杂度也会显著上升

因此 MVP 必须坚持该限制。

## 7.3 自动回收策略

为控制性能，live 节点应支持自动回收为冻结态。

建议触发条件：

- 用户点击了其他 HTML 节点
- 用户切换到普通节点或画布空白区域
- 节点缩放过小
- 节点移出视口很远
- 节点长时间无交互（后续可选）

## 7.4 视口与缩放策略

对于冻结态节点：

- 可作为普通贴图/纹理参与画布渲染
- 缩放小时不需要维持高成本交互能力

对于 live 节点：

- 仅当前可见且可操作时保持 live
- 过小缩放可自动退出 live

---

## 8. 技术路线

## 8.1 不自研 HTML/CSS/交互系统

本方案不建议自研：

- DOM
- CSS 布局
- 文本排版
- 按钮/输入框交互
- 滚动与焦点系统

这些都尽量交给浏览器/WebView 能力处理。

## 8.2 复用浏览器逻辑

推荐方向：

- HTML 节点底层依赖真实浏览器或 WebView 能力
- live 节点时使用真实交互能力
- 非 live 节点时使用冻结结果

也就是说：

- 交互逻辑复用浏览器
- 性能控制由宿主应用决定

## 8.3 首版不追求完整前端平台

MVP 不应承诺：

- 任意复杂网页应用都稳定运行
- 多个复杂动态网页同时交互
- 完整浏览器级事件桥与脚本桥

首版目标应聚焦于：

- 常见 HTML/CSS 片段
- 轻量交互控件
- 内部按钮/输入/滚动的直接操作

---

## 9. JavaScript 与动态能力

虽然“交互”是高优先级，但这不等于第一版要开放完整 JS 平台能力。

### 9.1 第一版建议

第一版优先保证：

- HTML/CSS 的视觉呈现
- 按钮点击
- 输入框交互
- 滚动交互
- 基础链接与 hover

### 9.2 第一版不建议主动承诺

- 任意用户脚本执行
- 框架级前端应用兼容
- 宿主高权限 JS Bridge

原因：

- 这些能力会显著增加安全、维护和调试成本
- 与“性能 + 低复杂度”的目标冲突

如果底层 WebView 不可避免会运行少量页面自身脚本，也应在产品定义上保持保守，不把“完整 JS 支持”作为首版卖点。

---

## 10. 资源与兼容性边界

### 10.1 支持目标

首版目标是支持：

- 常见 HTML 组件片段
- 内部面板、卡片、按钮组、输入框、说明块
- 轻量 CSS 样式

### 10.2 不承诺事项

首版不承诺：

- 任意外部完整网页都稳定可用
- 任意第三方前端项目直接无改造运行
- 复杂依赖网络资源的页面稳定加载

### 10.3 资源建议

首版优先支持：

- 本地资源
- 项目内导入资源
- 基础 CSS

远程资源、外链字体、复杂外部依赖建议延后处理。

---

## 11. 与当前项目的落地建议

基于当前代码结构，建议新增独立 Html 节点类型，而不是复用 Text 节点。

### 11.1 数据层

文件：`src/model.rs`

建议新增：

- `NodeKind::Html`
- `NodeData::Html { html_source: String, ... }`

建议补充的运行时状态（不一定全部持久化）：

- 是否为当前 live 节点
- 冻结图像/纹理引用
- 最后一次可见画面缓存
- 渲染错误状态

### 11.2 节点创建

文件：`src/app/nodes.rs`

建议新增：

- `create_html_node(pos: Pos2) -> usize`

默认尺寸建议略大于 Text 节点，例如：

- `420 x 260`

### 11.3 渲染层

文件：`src/app/ui/canvas_nodes_render.rs`

建议新增：

- `NodeKind::Html` 分支
- Frozen 态显示冻结结果
- Live 态显示真实交互网页区域
- 当前 live 节点高亮/状态标识（可选）

### 11.4 编辑器层

文件：`src/app/ui/canvas_editors.rs`

建议新增：

- HTML 源码编辑器
- 切换/关闭 live 的控制入口
- 基础状态提示

### 11.5 运行时管理模块

建议新增独立模块，例如：

- `src/app/html_runtime.rs`

负责：

- live 节点生命周期管理
- Frozen / Live 切换
- 单 live 节点约束
- 首次点击透传
- 冻结结果更新
- 失败与回收状态管理

### 11.6 持久化

文件：`src/app/persistence.rs`

建议新增：

- Html 节点内容保存/加载
- 必要元数据持久化

### 11.7 自动化协议

文件：`src/app/automation.rs`

建议新增：

- `kind = "html"`
- 允许自动化创建与更新 Html 节点

## 12. MVP 推荐实现顺序

## Phase 1：基础节点与冻结显示

目标：先接入 Html 节点基本数据流。

实现：

- `NodeKind::Html`
- `NodeData::Html`
- 创建节点
- 编辑 HTML 内容
- 保存/加载
- Frozen 态显示占位或基础冻结画面

## Phase 2：单 live 节点能力

目标：让节点真正可交互。

实现：

- 同时仅一个 live 节点
- 点击节点内部自动进入 live
- 旧 live 节点自动冻结
- live 节点能接收真实交互

## Phase 3：首次点击透传

目标：解决“按钮必须直接可点”。

实现：

- 点击即活
- 节点切换为 live 后，本次点击继续传给网页内容
- 用户不感知中间切换步骤

## Phase 4：回收与优化

目标：控制性能和体验稳定性。

实现：

- 缩放过小自动冻结
- 移出视口自动冻结
- 错误状态显示
- 资源回收与复用

---

## 13. 明确不做的内容（MVP）

以下能力建议明确延后：

- 多个 HTML 节点同时 live
- 完整 JS 平台开放
- 宿主系统高权限桥接
- 多网页节点复杂联动
- 把 HTML 节点做成完整浏览器工作台

原因：

- 都会显著拉高复杂度和维护成本
- 与当前目标不匹配

---

## 14. 风险点

这个方案虽然是当前最平衡的，但仍有几个关键难点：

### 14.1 首次点击透传

这是体验是否成立的关键技术点。

### 14.2 Live 与 Frozen 切换

需要保证切换时视觉和交互尽量平滑。

### 14.3 坐标与输入同步

live 网页区域必须和 Canvas 节点位置、尺寸保持一致。

### 14.4 焦点管理

键盘输入、鼠标事件、画布操作之间需要明确优先级。

这些都是实现中的重点风险，需要优先验证。

---

## 15. 验收标准（MVP）

满足以下条件即可认为 MVP 合格：

1. 用户可以创建 Html 节点
2. 用户可以编辑并保存 HTML 内容
3. 节点中的按钮可以直接点击，不需要先双击进入
4. 节点中的输入区可以直接获得焦点并输入
5. 节点中的滚动区域可以直接滚动
6. 同时最多只有一个 HTML 节点 live
7. 切换到新的 HTML 节点时，旧节点会自动冻结
8. 多个 HTML 节点同时存在时，画布仍可基本流畅操作
9. 缩放/移出视口后，live 节点可自动回收为冻结态
10. app 不因 HTML 节点切换导致明显卡死或崩溃

---

## 16. 一句话决策

HTML 节点第一版采用：

**“直接可交互、点击即活、首次点击透传、同一时刻仅一个 live 节点、其他节点冻结”的方案，在保证按钮可直接点击的前提下，尽量平衡性能、交互体验和维护成本。**
